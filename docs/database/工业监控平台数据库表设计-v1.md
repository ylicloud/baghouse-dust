# 工业监控平台数据库表设计 v1

## 1. 设计目标

本版本数据库设计用于支撑以下目标：

- 支撑 Kepware OPC UA -> BridgeService -> 平台库 的主链路。
- 满足设备总览、报警、趋势、报表、审计、手机查看等第一阶段功能。
- 兼顾多项目复制能力，避免平台深度依赖 WinCC 内部数据库。
- 结构尽量清晰，便于后续升级维护。

数据库建议使用 **SQL Server 2022**。

## 2. 表分层

建议将表分为 5 类：

1. 基础配置表
2. 实时与历史数据表
3. 报警表
4. 用户权限与审计表
5. 报表与同步控制表

## 3. 基础配置表

### 3.1 PlatformSites

用途：定义站点或项目实例，用于多项目复制时区分数据归属。

关键字段建议：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| Id | bigint PK | 主键 |
| SiteCode | nvarchar(64) | 站点编码，唯一 |
| SiteName | nvarchar(128) | 站点名称 |
| TimeZoneId | nvarchar(64) | 时区标识 |
| IsEnabled | bit | 是否启用 |
| CreatedAt | datetime2 | 创建时间 |
| UpdatedAt | datetime2 | 更新时间 |

索引建议：

- `UX_PlatformSites_SiteCode`

### 3.2 Devices

用途：定义设备、PLC 或逻辑设备。

关键字段建议：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| Id | bigint PK | 主键 |
| SiteId | bigint FK | 所属站点 |
| DeviceCode | nvarchar(64) | 设备编码，站点内唯一 |
| DeviceName | nvarchar(128) | 设备名称 |
| DeviceType | nvarchar(64) | 设备类型，如 PLC、Fan、DustCollector |
| SourceType | nvarchar(32) | 数据源类型，如 opcua |
| SourceEndpoint | nvarchar(256) | OPC UA 地址或数据源地址 |
| IsEnabled | bit | 是否启用 |
| CreatedAt | datetime2 | 创建时间 |
| UpdatedAt | datetime2 | 更新时间 |

索引建议：

- `UX_Devices_SiteId_DeviceCode`
- `IX_Devices_SiteId_IsEnabled`

### 3.3 TagDefinitions

用途：定义平台级点位，不直接把点位意义散落在代码中。

关键字段建议：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| Id | bigint PK | 主键 |
| DeviceId | bigint FK | 所属设备 |
| TagCode | nvarchar(128) | 平台点位编码 |
| TagName | nvarchar(128) | 点位名称 |
| SourceNode | nvarchar(256) | OPC UA 节点标识 |
| DataType | nvarchar(32) | bool、int、double、string 等 |
| Unit | nvarchar(32) | 单位 |
| PrecisionScale | int | 显示精度 |
| ArchiveEnabled | bit | 是否入历史 |
| SnapshotIntervalSeconds | int | 快照周期 |
| Deadband | decimal(18,6) null | 变化阈值 |
| IsWritable | bit | 是否允许平台写入 |
| IsAlarmSource | bit | 是否参与平台报警 |
| DisplayOrder | int | 展示顺序 |
| IsEnabled | bit | 是否启用 |
| CreatedAt | datetime2 | 创建时间 |
| UpdatedAt | datetime2 | 更新时间 |

索引建议：

- `UX_TagDefinitions_DeviceId_TagCode`
- `IX_TagDefinitions_DeviceId_IsEnabled`
- `IX_TagDefinitions_IsAlarmSource_IsEnabled`

### 3.4 DeviceGroups

用途：用于页面总览和统计聚合，比如“除尘器一组”“风机组”“1号线设备”。

关键字段建议：

## 4. 实时与历史数据表

### 4.1 TagSnapshots

用途：保存每个点位的最新状态，供总览页和手机首页快速读取。

建议：一条点位仅保留一行最新值，由 BridgeService 做 upsert。

关键字段建议：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| TagId | bigint PK/FK | 对应 TagDefinitions.Id |
| ValueText | nvarchar(256) | 字符串形式值 |
| ValueNumber | decimal(24,8) null | 数值型值 |
| ValueBool | bit null | 布尔型值 |
| QualityCode | nvarchar(32) | 质量码 |
| SourceTimestamp | datetime2 | 数据源时间 |
| ReceivedAt | datetime2 | Bridge 收到时间 |
| UpdatedAt | datetime2 | 快照更新时间 |

索引建议：

- 主键即 `TagId`
- `IX_TagSnapshots_UpdatedAt`

### 4.2 TagHistory

用途：存储历史趋势数据。

建议：

- 仅保存需要归档的点位。
- 采用按时间和点位查询友好的索引。

关键字段建议：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| Id | bigint PK | 主键 |
| TagId | bigint FK | 点位 |
| ValueText | nvarchar(256) | 字符串值 |
| ValueNumber | decimal(24,8) null | 数值值 |
| ValueBool | bit null | 布尔值 |
| QualityCode | nvarchar(32) | 质量码 |
| SourceTimestamp | datetime2 | 数据源时间 |
| ReceivedAt | datetime2 | 入库时间 |

索引建议：

- `IX_TagHistory_TagId_SourceTimestamp`
- `IX_TagHistory_SourceTimestamp`

如数据量增长明显，可按月分区或冷热分表。

### 4.3 DeviceStatusSnapshots

用途：保存设备级聚合状态，供设备总览和领导端快速展示。

关键字段建议：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| DeviceId | bigint PK/FK | 设备主键 |
| RunStatus | nvarchar(32) | 运行状态 |
| FaultStatus | nvarchar(32) | 故障状态 |
| AlarmLevel | nvarchar(32) | 当前最高报警等级 |
| IsOnline | bit | 是否在线 |
| LastSeenAt | datetime2 | 最后在线时间 |
| UpdatedAt | datetime2 | 更新时间 |

索引建议：

- 主键即 `DeviceId`
- `IX_DeviceStatusSnapshots_IsOnline_UpdatedAt`

## 5. 报警表

### 5.1 AlarmDefinitions

用途：定义平台级报警规则，不直接复刻 WinCC 全量报警模型，而是定义平台关心的规则。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| Id | bigint PK | 主键 |
| SiteId | bigint FK | 所属站点 |

- `IX_AlarmDefinitions_DeviceId_IsEnabled`

### 5.2 AlarmEvents

用途：保存报警产生、恢复、确认等事件。

## 8. 第一阶段最小可落地表集

如果第一阶段控制复杂度，建议先落以下核心表：

- `PlatformSites`
- `Devices`
- `TagDefinitions`
- `TagSnapshots`
- `TagHistory`
- `AlarmDefinitions`
- `AlarmEvents`
- `Users`
- `Roles`
- `UserRoles`
- `AuditLogs`
- `SyncCheckpoints`

其余表可在第二阶段补充。

## 9. 字段设计建议

### 9.1 时间字段

统一使用 `datetime2`，并尽量按 UTC 存储，再在页面层做时区转换。

### 9.2 主键策略

第一阶段建议使用 `bigint identity`，简单直接，易于维护。

### 9.3 金额以外数值

模拟量建议使用：

- `decimal(24,8)` 用于高精度数据
- 或 `float` 用于对性能要求高且误差可接受的趋势数据

第一阶段更建议统一使用 `decimal(24,8)`，减少歧义。

### 9.4 扩展字段

涉及规则、报表定义、附加明细时，优先保留：

- `DefinitionJson`
- `Detail`

避免第一阶段把表拆得过细。

## 10. 推荐索引策略

第一阶段优先保证以下查询：

1. 根据站点、设备查看最新状态
2. 根据点位和时间区间查趋势
3. 根据时间和状态查报警
4. 根据用户和时间查审计

因此索引重点放在：

- `TagHistory(TagId, SourceTimestamp)`
- `AlarmEvents(Status, TriggeredAt)`
- `AlarmEvents(DeviceId, TriggeredAt)`
- `AuditLogs(UserId, CreatedAt)`

## 11. 数据保留策略建议

建议从第一阶段就定义保留周期：

- `TagSnapshots`：长期保留最新值
- `TagHistory`：明细保留 6 到 12 个月，之后归档
- `AlarmEvents`：长期保留
- `AuditLogs`：长期保留或至少保留 24 个月
- `ReportExecutions`：按实际情况保留

## 12. 当前版本结论

数据库设计 v1 的核心思路是：

- 平台拥有自己的标准化数据库；
- BridgeService 负责从 Kepware OPC UA 同步数据入平台库；
- Web、手机、报表、报警都基于平台库工作；
- 项目差异主要体现在配置和基础定义表，而不是散落在代码里。
